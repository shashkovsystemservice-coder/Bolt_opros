# Survey Pro: Архитектурное видение и поэтапный план

Задача — навести архитектурный порядок в проекте Survey Pro, заложив фундамент для масштабирования, разделив создание инструментов, сбор данных и будущую аналитику.

### 1. Схема уровней системы

В основе новой архитектуры лежит простое и ясное разделение на три логических слоя:

| Уровень | Название | Сущность | Описание | Статус |
| :--- | :--- | :--- | :--- | :--- |
| **1. Instrument** | **Инструмент** | Шаблон/определение | **ЧТО используем для сбора данных.** Это форма, опросник, чек-лист. Неизменяемая структура вопросов. | Реализовано |
| **2. Data** | **Данные** | Ответы/заполнения | **ЧТО получили в результате.** Набор сырых, неинтерпретированных ответов от респондентов, привязанных к Инструменту. | Реализовано |
| **3. Analysis** | **Анализ** | Отчет/результат | **КАКИЕ выводы сделали.** Интерпретация собранных Данных на основе логики, заложенной в Инструменте (например, по формулам Blueprint). | Будущее |

### 2. Предложение по UI-структуре

Для отражения новой логики предлагаю минимально изменить навигацию и процесс создания.

**A. Боковая панель (Sidebar):**

| Текущая | Новая | Комментарий |
| :--- | :--- | :--- |
| `Создание` | (уходит в кнопку) | |
| `Опросы` | `Инструменты` | Список всех созданных опросников. |
| `Чек-листы` | (объединяется) | Становится частью Инструментов. |
| `Отчеты` | `Аналитика` `(скоро)` | Заглушка для будущего функционала. |
| `Данные` | `Данные` | |
| ↳ `Ответы` | ↳ `Ответы` | Все собранные данные (submissions). |
| ↳ `Контакты` | ↳ `Контакты` | Управление респондентами. |

**B. Процесс создания нового Инструмента:**

1.  Пользователь нажимает единую кнопку **«+ Создать инструмент»**.
2.  Появляется модальное окно или страница **«Выберите способ создания»** с карточками:
    *   **«С нуля»**: Ручное добавление вопросов. (Существующий функционал)
    *   **«AI-генератор»**: Простая генерация по теме. (Существующий функционал)
    *   **«Конструктор Blueprint»**: Создание сложного диагностического инструмента с осями и формулами. `(Advanced)`
    *   **«Импорт из файла»**: Восстановление функционала импорта. `(скоро)`

Эта структура явно разделяет способы создания и готовит пользователя к разным уровням сложности.

### 3. Терминологическая таблица

Ключ к порядку — консистентная и нейтральная терминология.

| Было (смешанные понятия) | Стало (нейтральный термин) | Обоснование |
| :--- | :--- | :--- |
| `Опрос`, `Чек-лист`, `Диагностика` | `Инструмент` | Единый термин для сущности, которая собирает данные. |
| `Создание нового опроса` | `Создание нового инструмента` | Процесс создания сущности первого уровня. |
| `Отчет` | `Аналитический отчет` / `Результат` | Термин для сущности третьего уровня (будущее). |
| `Ответы` / `Заполнения` | `Собранные данные` / `Ответы` | Сырые данные, полученные с помощью Инструмента. |

### 4. Пошаговый план (Roadmap)

Движемся итерационно, без поломок.

*   **Шаг 1: Рефакторинг терминологии и UI.**
    *   **Задача:** Переименовать элементы в боковой панели и ключевые заголовки (`Опросы` → `Инструменты` и т.д.).
    *   **Результат:** Пользователь начинает привыкать к новой логике без изменения функционала. Минимальный риск.

*   **Шаг 2: Централизация создания.**
    *   **Задача:** Создать единую точку входа («+ Создать инструмент») и экран выбора способа создания. Подключить к нему существующие ручной и AI-потоки.
    *   **Результат:** Потоки создания логически разделены, подготовлена почва для Blueprint.

*   **Шаг 3: Формализация Blueprint.**
    *   **Задача:** Выделить Meta-Survey в отдельный поток «Конструктор Blueprint» на экране выбора.
    *   **Результат:** Самый сложный способ создания изолирован и позиционируется как инструмент для экспертов.

*   **Шаг 4: Подготовка слоя "Аналитика".**
    *   **Задача:** Создать заглушку для раздела «Аналитика». Спроектировать (но не реализовывать) связь `Analysis` → `Instrument` в БД.
    *   **Результат:** Архитектурно и визуально система готова к внедрению аналитического движка.

### 5. Риски и их митигация

*   **Риск №1:** Негативная реакция пользователей на изменение привычных названий.
    *   **Митигация:** Проводить изменения в рамках одного релиза. Добавить ненавязчивые подсказки (tooltips) при первом входе, объясняющие, что «Опросы» теперь называются «Инструменты».

*   **Риск №2:** Архитектурный долг при "сшивке" старого UI с новой логикой.
    *   **Митигация:** Строго следовать предложенным шагам. Не пытаться реализовать все сразу. На первом этапе меняется только текст и навигация, без затрагивания логики компонентов.
